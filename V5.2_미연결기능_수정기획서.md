# SignFlow V5.2 미연결 기능 수정 기획서

## 채널(Channel) / 스크린월(Screen Wall) / 조건부 재생(Conditional Playback)

---

## 1. 문서 개요

V5.1까지의 수정으로 기본 콘텐츠 흐름(콘텐츠 → 플레이리스트 → 스케줄 → 디바이스 → 플레이어)은 정상 동작한다.
그러나 **채널**, **스크린월**, **조건부 재생** 세 기능은 DB 스키마와 백엔드 API, 프론트엔드 UI 페이지까지 구현되어 있으나
**실제 플레이어와 연결되지 않아 사용이 불가능한 상태**이다.

본 기획서는 세 기능 각각에 대해:
1. 설계 의도(이 기능이 왜 필요한가)
2. 현재 구현 현황(어디까지 되어 있는가)
3. 끊어진 지점(정확히 어디가 안 되는가)
4. 수정 계획(무엇을 어떻게 고칠 것인가)
5. 수정 대상 파일과 변경 상세

를 기술한다.

# Part A. 채널 (Channel)

## A-1. 설계 의도

채널은 **스케줄이 없을 때 재생되는 기본 콘텐츠 경로**이다.

### 사용 시나리오
```
[시나리오 1] 신규 매장 오픈 준비
  - 관리자가 "기본 채널"을 만들고 회사 소개 영상, 로고 이미지를 등록
  - 매장 디바이스를 채널에 할당
  - 스케줄 없이도 디바이스가 즉시 기본 콘텐츠 루프 재생

[시나리오 2] 스케줄 시간대 밖
  - 오전 9시~오후 6시 프로모션 스케줄이 종료되면
  - 플레이어가 자동으로 기본 채널 콘텐츠로 전환
  - 빈 화면(DefaultScreen) 대신 브랜드 콘텐츠가 표시됨

[시나리오 3] 실시간 콘텐츠 교체
  - 관리자가 채널 콘텐츠를 변경하면
  - 해당 채널에 할당된 모든 디바이스에 Socket.IO로 즉시 통지
  - 디바이스가 새 콘텐츠를 실시간으로 반영
```

### 콘텐츠 재생 우선순위
```
1순위: 긴급 메시지 (EmergencyOverlay) — 항상 최상위
2순위: 활성 스케줄 (시간/요일 조건 충족)
3순위: 기본 채널 (isDefault=true인 채널의 콘텐츠)
4순위: DefaultScreen (시계 화면 — 아무것도 없을 때)
```

## A-2. 현재 구현 현황

### DB 스키마 (완성)
```prisma
model Channel {
  id          String   @id @default(uuid())
  tenantId    String
  name        String
  description String?
  isDefault   Boolean  @default(false)   // 스케줄 없을 때 재생
  isActive    Boolean  @default(true)
  createdBy   String
  tenant      Tenant   @relation(fields: [tenantId])
  creator     User     @relation(fields: [createdBy])
  contents    ChannelContent[]
  devices     ChannelDevice[]
}

model ChannelContent {
  id        String  @id @default(uuid())
  channelId String
  contentId String
  order     Int         // 재생 순서
  duration  Int?        // 개별 재생 시간 오버라이드 (초)
  channel   Channel  @relation(fields: [channelId])
  content   Content  @relation(fields: [contentId])
}

model ChannelDevice {
  channelId  String
  deviceId   String
  assignedAt DateTime @default(now())
  channel    Channel  @relation(fields: [channelId])
  device     Device   @relation(fields: [deviceId])
  @@id([channelId, deviceId])
}
```

### 백엔드 API (완성 — 10개 엔드포인트)

| 메서드 | 경로 | 기능 | 권한 |
|--------|------|------|------|
| GET | `/api/channels` | 채널 목록 (검색, 페이지네이션) | VIEWER+ |
| GET | `/api/channels/:id` | 채널 상세 (콘텐츠+디바이스 포함) | VIEWER+ |
| POST | `/api/channels` | 채널 생성 | USER+ |
| PUT | `/api/channels/:id` | 채널 수정 | USER+ |
| DELETE | `/api/channels/:id` | 채널 삭제 | STORE_MANAGER+ |
| POST | `/api/channels/:id/contents` | 콘텐츠 추가 | USER+ |
| DELETE | `/api/channels/:id/contents/:itemId` | 콘텐츠 제거 | USER+ |
| PUT | `/api/channels/:id/contents/reorder` | 콘텐츠 순서 변경 | USER+ |

## A-3. 끊어진 지점 (5개)

### 끊김 1: 프론트엔드 — 디바이스 할당 UI 없음
- **현상**: ChannelsPage에서 채널에 할당된 디바이스 목록은 보이지만, 디바이스를 **추가/제거하는 UI가 없음**
- **원인**: `assignDevices()` API 메서드는 존재하나 호출하는 버튼/모달이 ChannelsPage에 없음
- **위치**: `frontend/src/pages/ChannelsPage.tsx`
- **영향**: 관리자가 채널에 디바이스를 할당할 수 없어 채널 자체가 무용

### 끊김 2: 플레이어 — 기본 채널 데이터 미수신
- **현상**: 플레이어가 스케줄만 fetch하고, 채널 정보는 전혀 요청하지 않음
- **원인**: `player/src/utils/api.ts`에 채널 관련 API 호출 없음
- **위치**: `player/src/utils/api.ts`, `player/src/hooks/useSchedule.ts`
- **코드 증거**:
  ```typescript

  if (active) {
    loadContentForSchedule(active)
  } else {
    setCurrentPlaylist(null)    // ← 여기서 채널 대신 null 설정
    setCurrentLayout(null)
  }
  ```

### 끊김 3: 플레이어 — 기본 채널 Fallback 렌더링 없음
- **현상**: 활성 스케줄이 없으면 DefaultScreen(시계 화면)만 표시
- **원인**: `player/src/App.tsx`의 렌더링 분기에 채널 재생 분기가 없음
- **위치**: `player/src/App.tsx`, `player/src/components/DefaultScreen.tsx`
- **현재 렌더링 분기**:
  ```
  currentLayout 있음  → ZoneLayoutPlayer
  currentPlaylist 있음 → PlaylistPlayer
  둘 다 null          → DefaultScreen (시계)  ← 여기에 채널 재생이 들어가야 함
  ```

### 끊김 4: 백엔드 — 디바이스 스케줄 API에 채널 미포함
- **현상**: `GET /api/devices/:id/schedules`가 스케줄만 반환하고 기본 채널 정보를 포함하지 않음
- **원인**: `playerController.js`의 `getDeviceSchedules()`가 Channel 테이블을 조회하지 않음
- **위치**: `backend/src/controllers/playerController.js`

### 끊김 5: Socket.IO — 채널 변경 실시간 알림 없음
- **현상**: 채널 콘텐츠를 추가/삭제/순서변경해도 디바이스에 알림이 가지 않음
- **원인**: `channelController.js` 200행에 TODO 주석만 존재
- **위치**: `backend/src/controllers/channelController.js`
- **코드 증거**:

### 수정 A-2: 백엔드 — 디바이스 스케줄 API에 기본 채널 포함

**파일**: `backend/src/controllers/playerController.js`

**변경 내용**:
`getDeviceSchedules()` 함수에서 스케줄 외에 디바이스의 기본 채널도 함께 반환

**변경 전** (현재):
```javascript
// getDeviceSchedules 반환값
{
  schedules: [ ...ScheduleEntry[] ]
}
```

**변경 후**:
```javascript
// getDeviceSchedules 반환값
{
  schedules: [ ...ScheduleEntry[] ],
  defaultChannel: {
    id: "채널ID",
    name: "기본 프로모션",
    items: [
      {
        id: "항목ID",
        content: {
          id: "콘텐츠ID",
          name: "회사소개.mp4",
          type: "VIDEO",
          fileUrl: "/uploads/...",
          duration: 60,
          canvasData: null   // CANVAS 타입일 경우 포함
        },
        duration: 60,       // 채널 레벨 오버라이드 또는 원본
        order: 1
      },
      // ...
    ]
  } | null   // 기본 채널이 없으면 null
}
```

**구현 로직**:
```javascript
async function getDeviceSchedules(req, res) {
  const { deviceId } = req.params;

  // 1. 기존 스케줄 조회 (현재 로직 유지)
  const schedules = await getSchedulesForDevice(deviceId);

  // 2. 디바이스의 기본 채널 조회 (신규)
  const channelDevice = await prisma.channelDevice.findFirst({
    where: { deviceId },
    include: {
      channel: {
        where: { isDefault: true, isActive: true },
        include: {
          contents: {
            orderBy: { order: 'asc' },
            include: {
              content: {
                select: {
                  id: true, name: true, type: true,
                  fileName: true, filePath: true, url: true,
                  duration: true, storageType: true, canvasJson: true,
                  thumbnail: true
                }
              }
            }
          }
        }
      }
    }
  });

  // 3. 채널 콘텐츠를 플레이어 형식으로 매핑
  let defaultChannel = null;
  if (channelDevice?.channel) {
    const ch = channelDevice.channel;
    defaultChannel = {
      id: ch.id,
      name: ch.name,
      items: ch.contents.map(cc => ({
        id: cc.id,
        content: mapContentForPlayer(cc.content),  // 기존 매핑 함수 재사용
        duration: cc.duration || cc.content.duration || 10,
        order: cc.order
      }))
    };
  }

  res.json({ schedules, defaultChannel });
}
```

```typescript
export async function fetchSchedules(deviceId: string) {
  const res = await api.get(`/devices/${deviceId}/schedules`);
  const { schedules, defaultChannel } = res.data;

  usePlayerStore.getState().setSchedules(schedules);
  usePlayerStore.getState().setDefaultChannel(defaultChannel);  // 신규

  return schedules;
}
```

# Part B. 스크린월 (Screen Wall)

## B-1. 설계 의도

스크린월은 **여러 대의 디스플레이를 하나의 큰 화면으로 결합**하는 기능이다.

### 사용 시나리오
```
[시나리오 1] 로비 비디오월
  - 3x2 = 6대의 55인치 TV를 벽에 배치
  - 하나의 4K 영상을 6개 화면에 분할 표시
  - 각 디바이스가 자기 위치(row, col)에 해당하는 영역만 확대 재생
  - 베젤(TV 테두리) 두께를 보정하여 영상이 자연스럽게 이어짐

[시나리오 2] 정보 디스플레이 월
  - 2x3 = 6대의 디스플레이
  - 전체를 하나의 대형 캔버스로 사용
  - 캔버스 에디터에서 5760x2160 해상도로 디자인

[시나리오 3] 동기화 재생
  - SyncGroup으로 묶인 디바이스들이 동일 시점에 콘텐츠 전환
  - LAN 모드: 200ms 이내 동기화 (같은 네트워크)
  - WAN 모드: 1000ms 이내 동기화 (원격)
```

### 핵심 개념: 베젤 보정
```
물리적 배치 (베젤 포함):
┌──────┐  ┌──────┐  ┌──────┐
│ TV 1 │▌▐│ TV 2 │▌▐│ TV 3 │   ▌▐ = 베젤 (테두리)
│      │▌▐│      │▌▐│      │
└──────┘  └──────┘  └──────┘

소프트웨어 보정 후 (시각적 결과):
┌────────────────────────────┐
│        연속된 하나의 영상      │
│                             │
└────────────────────────────┘

보정 원리:
  - 각 디바이스가 전체 영상을 scale(cols, rows)로 확대
  - 자기 위치만큼 translate로 이동
  - 베젤 두께(mm)를 픽셀로 환산하여 추가 오프셋
```

```javascript
// screenWallController.js — calculateBezelCompensation(wall, row, col)
// 입력: wall(행열+베젤mm+화면mm), row, col
// 출력: CSS transform 문자열

// 베젤 없는 경우:
transform = `scale(${cols}, ${rows}) translate(-${col*100/cols}%, -${row*100/rows}%)`

// 베젤 있는 경우:
bezelPxH = (bezelH / screenW) * 1920    // 베젤 mm → 픽셀 변환
bezelPxV = (bezelV / screenH) * 1080
offsetX = col * (1920 + bezelPxH * 2)    // 위치별 오프셋
offsetY = row * (1080 + bezelPxV * 2)
transform = `scale(${cols}, ${rows}) translate(-${offsetX}px, -${offsetY}px)`
```

# Part C. 조건부 재생 (Conditional Playback / Tag-based Playback)

## C-1. 설계 의도

조건부 재생은 **같은 스케줄이라도 디바이스 태그에 따라 다른 플레이리스트를 재생**하는 기능이다.

### 사용 시나리오
```
[시나리오 1] 매장 유형별 콘텐츠 분기
  스케줄: "점심 프로모션" (12:00~14:00, 전체 매장)
  조건:
    매장타입=카페  → "카페 메뉴 플레이리스트"     (우선순위 1)
    매장타입=식당  → "식당 메뉴 플레이리스트"     (우선순위 2)
    매칭 없음     → "기본 프로모션 플레이리스트"  (fallback)

  결과:
    카페 매장 디바이스 → 카페 메뉴 재생
    식당 매장 디바이스 → 식당 메뉴 재생
    기타 매장 디바이스 → 기본 프로모션 재생

[시나리오 2] 지역별 언어 분기
  스케줄: "공항 안내" (항상)
  조건:
    지역=국제선  → "영어 안내 플레이리스트"       (우선순위 1)
    지역=국내선  → "한국어 안내 플레이리스트"     (우선순위 2)

[시나리오 3] 층별 콘텐츠 분기
  스케줄: "빌딩 안내" (업무시간)
  조건:
    층=1층  → "로비 안내 플레이리스트"
    층=2층  → "사무실 안내 플레이리스트"
    층=B1  → "주차장 안내 플레이리스트"
```

### 태그 매칭 로직
```
입력:
  - 디바이스 태그: { "매장타입": "카페", "지역": "서울", "층": "1층" }
  - 스케줄 조건들 (우선순위 내림차순):
    [1] tagKey="매장타입", tagValue="카페"  → playlistId="카페메뉴"
    [2] tagKey="지역",    tagValue="서울"   → playlistId="서울지역"
    [3] tagKey="층",      tagValue="1층"    → playlistId="1층안내"

처리:
  1. 조건을 우선순위(priority) 내림차순으로 정렬
  2. 첫 번째 조건 확인: 디바이스.tags["매장타입"] === "카페"? → YES
  3. 매칭 성공! → "카페메뉴" 플레이리스트 사용 (나머지 조건 무시)

  만약 어떤 조건도 매칭되지 않으면:
  → 스케줄의 기본 플레이리스트(fallback) 사용
```

```typescript
function loadContentForSchedule(entry: ScheduleEntry) {
  const { deviceTags } = usePlayerStore.getState();

  // 조건부 재생: 조건이 있으면 태그 매칭
  if (entry.conditions && entry.conditions.length > 0 && deviceTags) {
    // 우선순위 내림차순으로 이미 정렬되어 있음
    for (const condition of entry.conditions) {
      const deviceValue = deviceTags[condition.tagKey];
      if (deviceValue === condition.tagValue && condition.playlist) {
        console.log(
          `[Schedule] Tag match: ${condition.tagKey}=${condition.tagValue}`,
          `→ Playlist: ${condition.playlist.name}`
        );
        setCurrentPlaylist(condition.playlist);
        setCurrentLayout(entry.layout || null);
        return;
      }
    }
    console.log('[Schedule] No tag match, using fallback playlist');
  }

  // fallback: 기본 플레이리스트
  if (entry.playlist) {
    setCurrentPlaylist(entry.playlist);
  }
  setCurrentLayout(entry.layout || null);
}
```

## D-3. 데이터 흐름 변경 (Before → After)

### 채널
```
Before:
  스케줄 없음 → DefaultScreen (시계) → 빈 화면

After:
  스케줄 없음 → 기본 채널 있는가?
    → YES: PlaylistPlayer로 채널 콘텐츠 루프 재생
    → NO:  DefaultScreen (시계)
```

### 스크린월
```
Before:
  디바이스 → 전체 화면에 콘텐츠 표시 (항상 100%)

After:
  디바이스 시작 → wall-info 조회
    → inWall=true: CSS transform 적용 (scale + translate + bezel 보정)
    → inWall=false: 기존과 동일 (전체 화면)
```

### 조건부 재생
```
Before:
  스케줄 활성 → 기본 플레이리스트 재생 (항상 동일)

After:
  스케줄 활성 → 조건 있는가?
    → YES: 디바이스 태그로 매칭
      → 매칭 성공: 조건의 플레이리스트 재생
      → 매칭 실패: 기본 플레이리스트 재생 (fallback)
    → NO: 기본 플레이리스트 재생
```

## D-4. 의존 관계

```
playerController.js (백엔드)
  ├── 채널: defaultChannel 반환
  ├── 조건부: conditions + deviceTags 반환
  └── (스크린월은 별도 엔드포인트)

playerStore.ts (플레이어)
  ├── defaultChannel  ← fetchSchedules()에서 설정
  ├── deviceTags      ← fetchSchedules()에서 설정
  └── wallInfo        ← fetchWallInfo()에서 설정

useSchedule.ts (플레이어)
  ├── 채널 fallback   ← defaultChannel 읽기
  └── 태그 매칭       ← deviceTags + entry.conditions 읽기

App.tsx (플레이어)
  └── 월 transform    ← wallInfo 읽기
```
