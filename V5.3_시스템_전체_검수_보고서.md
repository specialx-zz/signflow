## 3. Part A: 프로세스 플로우 검증

### Flow 1: 콘텐츠 → 플레이리스트 → 스케줄 → 디바이스 재생

```
[콘텐츠 업로드] → [플레이리스트에 추가] → [스케줄 생성] → [디바이스 배포] → [플레이어 재생]
     ✅              ✅                    ✅              ✅              ✅
```

**정상 동작:** 기본 재생 플로우는 정상
**문제 발견:**
- ❌ 콘텐츠 삭제 시 해당 콘텐츠가 포함된 PlaylistItem이 정리되지 않음 (H-17)
- ❌ 플레이리스트 삭제 시 스케줄의 playlistId 참조가 끊어짐 (H-16)
- ⚠️ 플레이리스트 아이템 재정렬에 소유권 검증 없음 → 다른 테넌트 데이터 조작 가능 (H-11)

### Flow 2: 채널 → 콘텐츠 배포 → 디바이스 실시간 재생

```
[채널 생성] → [콘텐츠 추가] → [디바이스 할당] → [Socket 알림] → [플레이어 채널 재생]
     ✅           ✅             ❌                ❌              ✅ (로직은 정상)
```

**문제 발견:**
- ❌ 디바이스 할당 모달이 `/api/api/devices` 이중 경로로 404 오류 (C-03)
- ❌ Socket.IO 알림이 `req.app.get('io')` vs `req.io` 불일치로 전달 안됨 (C-11)
- ❌ `removeContent`에서 null 체크 없이 삭제 시도 → 크래시 (C-08)

### Flow 3: 스크린 월 → 디바이스 배치 → 베젤 보정 재생

```
[스크린 월 생성] → [그리드 디바이스 배치] → [동기화 그룹] → [플레이어 월 재생]
      ✅              ❌                      ✅            ✅ (로직은 정상)
```

**문제 발견:**
- ❌ 디바이스 배치 모달도 `/api/api/devices` 이중 경로로 404 (C-04)
- ❌ 스크린 월 수정/삭제에 테넌트 검증 없음 → 다른 업체의 월 조작 가능 (H-04)
- ❌ 동기화 그룹 삭제에도 테넌트 검증 없음 (H-05)

### Flow 4: 조건부 재생 → 태그 매칭 → 대체 플레이리스트

```
[스케줄 조건 설정] → [디바이스 태그 설정] → [플레이어 태그 매칭] → [대체 재생]

```
[긴급 메시지 생성] → [Socket 전송] → [플레이어 오버레이 표시]

## 4. Part B: CRITICAL 이슈 (즉시 수정 필요)

### C-01/C-02: 긴급 메시지 시스템 완전 미작동

**영향:** 긴급 상황 시 디바이스에 메시지가 표시되지 않음
**원인 1:** `EmergencyOverlay.tsx`에서 `usePlayerStore()`의 `socket`을 읽지만, `playerStore.ts`에 `socket` 필드가 정의되어 있지 않음 → 항상 `undefined`
**원인 2:** `useSocket.ts`에 `emergency:message`, `emergency:dismiss` 이벤트 리스너가 등록되지 않음

```
EmergencyOverlay → usePlayerStore().socket → undefined → 리스너 등록 안됨
useSocket.ts → schedule:update ✅, channel:update ✅, emergency:* ❌
```

**수정 방향:**
- `useSocket.ts`에 emergency 이벤트 리스너 추가
- 수신된 메시지를 playerStore의 새로운 `emergencyMessage` 상태에 저장
- `EmergencyOverlay`가 store의 `emergencyMessage`를 구독하도록 변경

### C-03/C-04: 채널/스크린월 디바이스 조회 API 경로 오류

**영향:** 채널 디바이스 할당, 스크린월 디바이스 배치 기능 완전 불능
**원인:** `apiClient`의 `baseURL`이 이미 `/api`인데, 요청 경로에 다시 `/api/`를 붙임

```javascript
// ChannelsPage.tsx:106 (잘못된 코드)
apiClient.get('/api/devices', ...) // → 실제 요청: /api/api/devices (404)

// 올바른 코드
apiClient.get('/devices', ...)     // → 실제 요청: /api/devices
```

**수정 방향:** 두 파일에서 `/api/devices`를 `/devices`로 변경

### C-05: 웹훅 모달 렌더링 불가

**영향:** 웹훅 생성, 로그 조회 기능 완전 불능
**원인:** `<Modal>` 컴포넌트에 필수 prop `isOpen`이 전달되지 않음

```tsx
// WebhooksPage.tsx:151 (잘못된 코드)
<Modal title="웹훅 추가" onClose={...}>

// 올바른 코드
<Modal isOpen={showCreate} title="웹훅 추가" onClose={...}>
```

### C-06/C-07: 백엔드 컨트롤러 존재하지 않는 모델/필드 참조

**영향:** 콘텐츠 여정 조회, 일간/주간 통계 API 호출 시 500 에러
**원인:**
- `channelController.js:329` → `prisma.scheduleItem` 모델이 스키마에 없음
- `statsController.js` → `Statistics` 모델에 `playedAt`, `duration` 필드 없음

### C-09/C-10: 라우트 섀도잉

**영향:** 캔버스 템플릿 목록, 채널 콘텐츠 여정 API 접근 불가
**원인:** Express 라우트 등록 순서에서 `/:id` 패턴이 특정 경로보다 먼저 매칭됨

```javascript
// routes/canvas.js - 잘못된 순서
router.get('/:id', ...)          // line 16: 이것이 먼저 매칭
router.get('/templates/list', ...) // line 20: 도달 불가 ("templates"가 :id로 매칭됨)

// 올바른 순서
router.get('/templates/list', ...) // 구체적 경로 먼저
router.get('/:id', ...)           // 와일드카드 나중에
```

### C-11: Socket.IO 참조 불일치

**영향:** 채널 콘텐츠 변경 시 플레이어에 실시간 알림 전달 안됨
**원인:** `channelController.js`의 `notifyChannelDevices`가 `req.app.get('io')`를 사용하지만, 서버 설정에서 `io`를 `req.io`로 바인딩했을 가능성

### C-12: 설정 저장 미동작

**영향:** 시스템 설정 변경이 실제로 저장되지 않음
**원인:** `routes/settings.js`의 PUT 엔드포인트가 success 응답만 반환하고 데이터를 저장하는 로직이 없음

**파일:** `player/src/utils/cache.ts`
**영향:** 장시간 운영 시 플레이어 메모리 사용량 지속 증가 → 크래시
**원인:** `URL.createObjectURL(blob)` 호출 후 `URL.revokeObjectURL()` 미호출
**수정:** 새 Blob URL 생성 전 이전 URL revoke 처리

### H-02: fetchSchedules 레이스 컨디션

**파일:** `player/src/utils/api.ts:64-68`
**영향:** 동시 스케줄 요청 시 `defaultChannel`, `deviceTags`가 잘못된 값으로 덮어씌워질 수 있음

### H-03: 조건부 재생 우선순위 무시
