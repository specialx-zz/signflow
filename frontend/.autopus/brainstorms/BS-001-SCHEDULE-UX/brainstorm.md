# BS-001: Schedule Management UX Improvements

> Auto-generated by `/auto idea` | 2026-04-16

## What

The schedule management page (`/schedules`) has accumulated UX pain points that create daily friction for store managers and tenant admins. This brainstorm evaluates improvements to reduce user errors, speed up task completion, and support scale beyond 100 schedules.

## Why

- **Silent failures**: No form validation lets users create schedules with backwards date ranges or zero devices — these fail downstream with no feedback
- **Device assignment UX**: Native `<select multiple>` with Ctrl+click is the worst standard control for multi-selection — causes mis-assignments and frustration
- **Workflow fragmentation**: Condition editor only accessible from detail modal, not edit flow — forces a 3-step detour
- **Scale ceiling**: Hard 100-item cap with no pagination silently hides data
- **Technical debt**: `SchedulesPage.tsx` at 408 lines violates the 300-line hard limit

## Who

| Persona | Pain Level | Frequency |
|---------|-----------|-----------|
| Store Manager | High | Daily (20-50 schedules, 5-15 devices) |
| Tenant Admin | Medium | Weekly (overview, audit, bulk reviews) |
| Super Admin | Low | Rare (troubleshooting only) |

## When

### I1: Form Validation with Inline Error Feedback

**Consensus**: All 3 perspectives rank this #1 or tied #1.

**Problem**: Users create schedules with end date before start date, end time before start time, zero devices, and WEEKLY repeat with no days — all accepted silently.

**Solution**:
- Real-time inline validation blocking submission on invalid state
- Date/time range cross-validation (end ≥ start)
- Minimum 1 device gate with counter ("0/15 selected")
- WEEKLY mode: render day-of-week toggle chips (Mon-Sun), require ≥ 1 selected
- New `validateScheduleForm()` pure function in `utils/scheduleValidation.ts`

### I6: Conflict Dialog with Schedule Identification and Resolution

**Consensus**: UX designer #3 (impact 4), PM deferred (high effort).

### I7: Schedule Duplication

**Consensus**: PM #5 (business value 4), others acknowledge as productivity multiplier.

### I8: Cursor-Based Pagination

**Consensus**: All perspectives acknowledge importance but rate lower urgency.

**Problem**: Hard 100-item cap silently hides data. Most tenants may not hit this today, but it's a trust-damaging silent failure when they do.
